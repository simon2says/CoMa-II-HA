    \begin{aufgabe}
        Zunächst schreiben wir eine Funktion {\ttfamily buildTree}, die aus einer Menge $S$ einen Radixbaum $T$ generiert. Dafür iterieren wir in jedem Element $s$ in der Menge $S$ über jedes Zeichen und wandern in dem Baum nach links, falls das Zeichen eine 0 ist, und nach rechts, falls das Item eine 1 ist. Dabei kann es sein, dass wir zunächst die entsprechenden Wege erstellen müssen. Am Ende der Iteration über $s$ setzen wir den Wert des aktuellen Knotens auf das Element in der Menge, über das wir eben iteriert haben.
        
        \begin{algorithm}[H]
            \caption{\ttfamily buildTree(S)}
            \KwIn{Menge $S$, leerer Baum $T$ mit $T$.root $\neq$ None}
            \KwOut{Radixbaum $T$ für Menge $S$}
            \For{$s \in S$}{
                curr $\gets$ T.root\\
                \For{$c \in s$}{
                    \If{$c = 0$}{
                        \If{curr.left $=$ None}{
                            curr.left $\gets$ Node(None)
                        }
                        curr $\gets$ curr.left
                    }
                    \Else{
                        \If{curr.right = None}{
                            curr.right $\gets$ Node(None)
                        }
                        curr $\gets$ curr.right
                    }
                }
                curr.value $\gets$ c
            }
            \Return{$T$}
        \end{algorithm}
        Wir schreiben auch eine weitere Funktion, die alle Elemente in einem Radix-Baum in lexikographischer Reihenfolge zurückgibt.
        
        \begin{algorithm}[H]
            \caption{\ttfamily retEles(n)}
            \KwIn{Ein Knoten $n$ in einem Radixbaum}
            \KwOut{Der Teilbaum unter $n$ in lexikographischer Reihenfolge}
            \If{n = None}{
                \Return{}
            }
            print(n.value)\\\ttfamily
            retEles(n.left)\\
            retEles(n.right)
        \end{algorithm}
        \begin{algorithm}[H]
            \caption{Einen Radix-Baum nutzen, um Liste zu sortieren}
            $T \gets$ {\ttfamily buildTree(S)}\\
            {\ttfamily retEles(T.root)}
        \end{algorithm}
        \begin{theorem*}
            Die Laufzeit des Algorithmus liegt in $\Theta(n)$, wobei $n$ die Summe der Längen der Elemente in $S$ ist.
        \end{theorem*}
        \begin{proof}
            Wir zeigen separat, dass die Laufzeit der Funktionen {\ttfamily retELes} und {\ttfamily buildTree} linear in der Eingabegröße liegen.

            Für {\ttfamily retEles} ergibt sich: Für jeden Knoten wird genau ein rekursiver Aufruf durchgeführt. Die Laufzeit liegt damit in $\BigO(S) \leq \BigO(n)$.

            Bei {\ttfamily buildTree} durchlaufen wir jedes Zeichen jedes Elements in zwei geschachtelten {\ttfamily for}-Schleifen. In diesen {\ttfamily for}-Schleifen geschehen lediglich Dinge in konstanter Zeit. Daraus ergibt sich eine Laufzeit von $\BigO(n)$, da $n$ als Summe aller Längen aller Elemente in $S$ definiert war. 
        \end{proof}
        \begin{theorem*}
            Der Algorithmus ist korrekt.
        \end{theorem*}
        \begin{proof}
            Jeder Eintrag wird an seiner korrekten Stelle untergebracht: Falls das aktuelle Zeichen eine $0$ ist, geht der Algorithmus nach links und baut dort möglicherweise einen Knoten. Falls das aktuelle Zeichen eine $1$ ist, geschieht das auf der anderen Seite. Der Algorithus wandert so lange in die korrekte Richtung, bis das aktuelle Element keine weiteren Zeichen enthält, und setzt den Wert des Knotens, bei dem es nun ist, auf das Element. Diese Prozedur wird für alle Elemente in $S$ durchgeführt.
        \end{proof}
    \end{aufgabe}

                \caption{{\ttfamily Inorder-Tree-Walk(x)} (iterativ)}
                \KwIn{Knoten $x$ in einem Suchbaum $T$, leerer Stack $S$}
                \KwOut{In-Order-Traversierung des Teilbaumes von $T$ unter $x$}

                \While{!{\ttfamily S.isEmpty()} or {\ttfamily curr.left} $\neq$ None or {\ttfamily curr.right} $\neq$ None}{
                    \uIf{{\ttfamily curr.left} $\neq$ None}{
                        {\ttfamily S.push}(curr)\\
                        curr $\gets$ {\ttfamily curr.left}
                    }
                    \Else{
                        print(curr)\\
                        \uIf{{\ttfamily curr.right} $\neq$ None}{

                        }

                            curr $\gets$ {\ttfamily S.pop}()\\
                            print(curr)\\
                            curr $\gets$ {\ttfamily curr.right}
                        }
                    }
                }
                print(curr)
            \end{algorithm}
        \begin{enumerate}[label=]
            \item 
            \begin{theorem*}
                Der Algorithmus ist korrekt.
            \end{theorem*}
            \begin{proof}
                Der Knoten {\ttfamily curr} ist so lange ein Knoten mit Kindern, wie es noch unbesuchte Knoten mit Kindern gibt: Falls es ein linkes Kind gibt, wird der Knoten so lange im Stack gespeichert, wie es noch weitere Knoten unter diesem Knoten gibt. Danach wird der Knoten geprintet und {\ttfamily curr} auf seinen aktuellen Kindknoten gesetzt. Falls es keinen linken Kindknoten gibt, wird {\ttfamily curr} stattdessen auf den rechten Kindknoten gesetzt. Damit wird eine korrekte Abarbeitung der Knoten gewährleistet und alle Knoten abgearbeitet.
            \end{proof}
        \end{enumerate}

            \item Wir zeigen per Induktion: Sei $k$ die Anzahl der während des gesamten Durchlaufs des Algorithmus besuchten Knoten und $n$ die gesamte Anzahl von Knoten in dem Baum. Dann gilt: $\exists \alpha, c\in \N: k \leq \alpha \cdot n + c $.

            \begin{proof}[Beweis durch starke Induktion über die Anzahl der Knoten.]
                Induktionsanfang: Für $n = 1$, $n = 2$ gilt die Aussage. Der Algorithmus wandert für Knotenanzahl $1$ nur zum kleinsten Knoten des Baumes und geht dort nicht weiter. Für Knotenanzahl $2$ geht der Algorithmus zunächst zum kleineren der beiden Knoten und von dort aus zu dessen Nachfolger für insgesamt im schlechtesten Fall $3 = 1\cdot n$ Knotenbesuche.

                Induktionsvoraussetzung: Für alle Bäume der Knotenanzahl $j: 0 \leq j < n$ gilt: Der Algorithmus besucht auf dem Baum höchstens $\alpha\cdot j + c$ Knoten.

                Induktionsschritt: Dann gilt auch: Der Algorithmus besucht auf dem Baum höchstens $\alpha\cdot n + c$ Knoten. Ohne Beschränkung der Allgemeinheit gehen wir davon aus, dass wir in einen der Bäume aus der Induktionsvoraussetzung ein Blatt einfügen, da für diesen Beweis lediglich die Struktur des Baumes relevant ist. Das Einfügen eines Blattes in einen Baum kann die Anzahl der besuchten Knoten um höchstens zwei erhöhen: Da wir ein Blatt eingefügt haben, ist der Knoten jedenfalls Baumnachfolger einer höheren Ebene. 
                
                Falls es sich um den kleinsten Knoten im Baum handelt, so ändert sich die Länge der Suche nach {\ttfamily Tree-Minimum(T.root)} um 1. Danach benötigen wir noch einen weiteren zusätzlichen Knotenbesuch, um den Nachfolger dieses Knotens zu finden. Dieser ist hier in jedem Fall sein direkter Elternknoten. Danach hat sich nichts weiteres an dem Baum verändert, er benötigt weiterhin so viele Knotenbesuche wie in der vorigen Knotenanzahl.

                Falls es sich nicht um den kleinsten Knoten im Baum handelt, handelt es sich um einen Knoten, der Nachfolger eines anderen Knotens ist. Dann benötigen wir zusätzliche Zeit, um diesen Nachfolger zu finden. Ist er das linke Kind eines Knotens, war der vorherige Nachfolger des Knotens, dessen Nachfolger er ist, sein Elternknoten. Dann benötigen wir wieder einen zusätzlichen Knotenbesuch, um ihn zu finden, und einen weiteren, um zu seinem Elternknoten zurückzukehren. Ist er stattdessen das rechte Kind eines Knotens, so ist er höchstens Nachfolger seines Elternknotens, da er ein Blattknoten ist. Damit benötigen wir einen weiteren Knotenbesuch, um ihn selbst zu finden, und einen Knotenbesuch, um zu seinem Elternknoten zurückzukehren, um dort nach seinem Nachfolger zu sehen. Auch hier haben wir nicht mehr als zwei zusätzliche Knotenbesuche im Vergleich zur vorherigen Knotenanzahl.
            \end{proof}
        \end{enumerate}

                2 \geq \Phi \cdot 1

                4 \geq \Phi \cdot 2

                    T(k) \geq \Phi\cdot T(k - 1).

                    T(n) \geq \Phi\cdot T(n - 1).

                    &\geq \Phi\cdot T(n - 1)

        \begin{enumerate}[label=(\alph*)]
            \item Es kommen höchstens $k$ Bitflips in einer beliebigen Iteration vor. Dies ist genau dann der Fall, wenn alle Bits flippen müssen, damit Marcel den Binärstring um jeweils $1$ erhöhen möchte. Für den Fall, dass nur das führende Bit $0$ ist und alle folgenden Bits $1$ sind, ändert sich jedes Bit, sofern Marcel den Bitstring durch eine Iteration erhöhen will. Da es insgesamt k Bits gibt, ändern sich somit alle Bits. Mehr als k Bits existieren nicht, womit sich in einer Iteration auch nicht mehr ändern können. Das Beispiel ist also $011...111$, für k Bits. 
            \item Sei $\Phi$ eine Potentialfunktion, wobei wir die als 
            \begin{align*}
                \Phi(D_i) = \text{Die Anzahl der Einsen im Bit}
            \end{align*}
            festlegen. Sofern nur ein Bitflip stattfindet von $0$ auf $1$ oder $1$ auf $0$, ist $c_i= 1$ und für $0$ auf $1$ ist 
            \begin{align*}
                \Phi(D_i) = \Phi(D_{i-1}) +1.
            \end{align*} Findet ein Bitflip von $1$ auf $0$ statt, so gilt 
            \begin{align*}
                \Phi(D_i)= \Phi(D_{i-1}) - 1.
            \end{align*} Damit ergibt sich für 
            \begin{align*}
                \Phi(D_i) - \Phi(D_{i-1}) = \begin{cases}
                    \Phi(D_{i-1}) +1 - \Phi(D_{i-1}) = 1 & \text{ Bitflip von $0$ auf $1$} \\
                    \Phi(D_{i-1}) -1 - \Phi(D_{i-1}) = -1 & \text{ Bitflip von $1$ auf $0$}
                \end{cases}
            \end{align*}
            Somit ist Für den Fall, dass ein Bitflip stattfindet, also $c_i = 1$ 
            \begin{align*}
                \hat{c_i}= 1 -1 = 0 \leq 2 \quad \text{ oder } \quad \hat{c_i}= 1 +1 = 2 \leq 2.
            \end{align*} Angenommen es gäbe $n \leq k$ Bitflips (da es maximal k Bitflips in einer Iteration geben kann, siehe Teil eins der Aufgabe), wobei $n \in \mathbb{N}$. Dann gilt für $\Phi(D_i)$ mit $n= m + o$, wobei $m$ alle Bitflips von $1$ auf $0$ und $o$ alle Bitflips von $0$ auf $1$, dass
            \begin{align*}
                \Phi(D_i) = \Phi(D_{i-1}) + o - m.
            \end{align*}
            Damit ergibt sich für 
            \begin{align*}
                \Phi(D_i) - \Phi(D_{i-1}) = \Phi(D_{i-1}) + o - m - \Phi(D_{i-1}) = o - m.
            \end{align*} Dann folgt für $\hat{c_i}$, dass 
            \begin{align*}
                \hat{c_i}=n +o - m=  o + m + o - m = 2 o \leq 2.
            \end{align*} Dies gilt, da im Maximalfall, wenn alle $1$ auf $0$ flippen, nach Teilaufgabe eins maximal ein Bit auf $1$ flippt, womit $o \leq 1$ gilt und die Voraussetzung erfüllt ist. 
            \item Mit Teil b gilt, dass für ein beliebiges n die ersten n Iterationen des Verfahrens mit Startstring $000...000$ eine obere Schranke an Bitflips benötigen. Allgemein gilt für jede Iteration, dass die Anzahl der Bitflips maximal $n$ sein kann. Da mit Teil zwei für jede Iteration $\hat{c_i}= c_i + \Phi(D_i) - \Phi(D_{i+1}) \leq 2$ mit $c_i$ als Anzahl der Bitflips gilt, folgt, dass die Summe aller Iterationen gegeben ist mit 
            \begin{align*}
                \sum_{i=1}^n \hat{c_i} = \sum_{i=1}^n c_i + \Phi(D_i) - \Phi(D_{i+1}) \leq \sum_{i=1}^n 2.
            \end{align*}
            Weiter ist $\sum_{i=1}^n \Phi(D_i) - \Phi(D_{i-1})$ eine Teleskopsumme und mit dem zweiten Teil ist $\Phi(D_0)=0$, sodass gilt:
            \begin{align*}
                \sum_{i=1}^n \Phi(D_i) - \Phi(D_{i+1}) = \Phi(D_n) - \Phi(D_0) = \Phi (D_n).
            \end{align*} Damit ergibt sich mit $\Phi(D_i)\geq 0$ und $\hat{c_i}< 2$:
            \begin{align*}
                \sum_{i=1}^n \hat{c_i} = \sum_{i=1}^n c_i + \Phi(D_i) - \Phi(D_{i+1}) = \sum_{i=1}^n c_i + \Phi(D_n) \leq 2n
            \end{align*} und somit, dass 
            \begin{align*}
                \sum_{i=1}^n c_i \leq 2n
            \end{align*} gilt. Damit ist die Anzahl der Bitflips der ersten $n$ Iterationen auf $2n$ beschränkt. Es gilt bei $n$ Iterationen mit der oberen Schranke aus dem zweiten Teil, dass 
            \begin{align*}
                \sum_{i=1}^n c_i \leq \sum_{i=1}^n \hat{c_i} \leq 2n. 
            \end{align*}
        \end{enumerate}
